# Proposal: 热量缺口管理工具

> **Change**: calorie-tracker
> **版本**: v1.0

---

## 1. 产品定位

个人单机版 Web 热量缺口管理工具。核心目标是帮用户**制造热量缺口**——基于用户的基础代谢和目标缺口，动态计算每日饮食预算，通过记录饮食摄入来监控是否处于缺口状态。

> **一句话**：设目标 → 自动算预算 → 记录吃了啥 → 看缺口达成没

---

## 2. 核心概念

```
热量缺口 = 总消耗（基础代谢 + 活动消耗）- 饮食摄入

  ↓                     ↓                      ↓
自动计算               用户设置               用户记录
（BMR 公式）          （活动系数）            （食物+克数）
```

---

## 3. 用户角色

| 角色 | 描述 | 核心需求 |
|------|------|---------|
| **减脂用户** | 有明确的减重目标 | 每日保持热量缺口来逐步减脂 |
| **体重管理者** | 希望维持当前体重 | 热量摄入 ≈ 总消耗 |
| **增肌用户** | 需要热量盈余（反向缺口） | 增重的同时控制质量 |

---

## 4. 核心目标

1. **自动算预算**：基于个人信息自动计算每日饮食预算，而非手动硬填一个数字
2. **缺口可视化**：每餐记录的摄入 vs 预算 = 当前缺口多少，一目了然
3. **趋势可追踪**：日/周/月维度的缺口达成情况，知道自己是否走在正确轨道上

## 6. 用户流程

```
首页（今日概览）
  │
  ├─ 记录 Tab ──┬─ AI 描述模式：输入自然语言 → AI 拆解 → 预览确认 → 添加记录
  │             │
  │             └─ 手动搜索模式：搜索食物 → 选择食物 → 输入克数 → 添加记录
  │                    │
  │                    └── 如果找不到 → 跳转食物库 → 新增自定义食物
  │
  ├─ 食物库 Tab ──→ 分类浏览 / 搜索 → 查看食物详情
  │                    │
  │                    └── 新增自定义食物
  │
  ├─ 统计 Tab ──→ 今日分布 / 七日趋势 / 月视图
  │
  └─ 设置 Tab ──┬─ 个人信息 / 活动系数 / 减重目标
                   ├─ 体重记录
                   └─ 大模型 API 配置（API 地址 / Key / 模型 / 模式切换）
```

## 7. 技术约束

| 项目 | 决策 |
|------|------|
| 架构 | 纯前端 SPA，无后端 |
| 存储 | 浏览器 LocalStorage |
| LLM 集成 | 支持 OpenAI 兼容 API（DeepSeek / Qwen / GPT） |
| 离线 | 100% 离线可用（大模型估算需网络） |
| 部署 | GitHub Pages / Vercel / Netlify |

## 9. 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM API CORS 限制 | 部分 API 不支持浏览器直接调用 | 文档注明 CORS 要求，推荐已知支持的 API 提供商 |
| LocalStorage 5MB 上限 | 长期使用可能接近上限 | 数据量预估<2MB/年，v1.2 考虑数据导出清理 |
| 规则引擎准确率 | 复杂描述识别不准 | 混合模式兜底，用户可手动调优 |
| 计算结果科学性 | 用户质疑公式准确性 | 展示计算过程公式，注明 Mifflin-St Jeor 公式来源 |
