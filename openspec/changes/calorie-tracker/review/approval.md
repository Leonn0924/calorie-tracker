## 二、阻塞问题（必须解决后才能开发）

### 🔴 BLOCK-01: 原型 v2 新增功能未纳入 specs

**问题**: 原型 v2 新增了三个重要模块，但 specs 中完全缺失：

1. **运动消耗模块** — 记录页新增运动板块
2. **体重追踪** — 统计页新增体重趋势图
3. **营养素维度** — 食物库增加蛋白/碳水/脂肪

**影响**: 如果按当前 specs 开发，原型与实现将严重不一致。

**解决方案**:
- **方案 A**（推荐）: 将运动消耗/体重追踪纳入 v1.0 MVP specs，营养素延后到 v1.1
- **方案 B**: 全部延后到 v1.1，v1.0 严格按当前 specs 开发
- **方案 C**: 全部纳入 v1.0，但开发工时需增加约 40%

**研发建议**: 选择方案 A — 运动消耗和体重追踪是核心闭环的一部分，不纳入会导致产品不完整。营养素可延后。

### 🔴 BLOCK-02: 内置食物库缺少营养素数据

**问题**: specs 中食物库首批 20+ 食物只定义了 `caloriesPer100g`，但如果 v1.0 需要展示营养素（即使延后），内置食物数据也需要同步补充。

**影响**: 即使延后到 v1.1，数据补充工作量不变。

**解决方案**: 开发时先只实现 calories，数据模型预留 protein/carbs/fat 字段，v1.1 时补充数据。

## 三、技术建议（非阻塞）

### 💡 建议-01: 存储层抽象

建议封装统一的存储层接口，便于后续扩展：

```typescript
interface Storage {
  get<T>(key: string): T | null
  set<T>(key: string, value: T): void
  remove(key: string): void
}
```

当前实现 LocalStorage，后续可无感切换到 IndexedDB。

### 💡 建议-02: 日期处理

建议使用 `date-fns` 或 `dayjs` 处理日期，避免原生 Date 的时区问题。尤其是跨日记录和日历展示。

### 💡 建议-03: 搜索性能

200ms 内的搜索要求对于 200+ 食物库是可行的（内存过滤），但如果扩展到 1000+ 食物，建议加入防抖 + 索引。

### 💡 建议-04: AI 估算缓存

建议对相同描述的 AI 估算结果做本地缓存（LRU），减少 API 调用次数和成本。
