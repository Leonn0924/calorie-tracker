# PRD：热量缺口管理工具 v1.0

## 1. 产品概述

### 1.1 产品定位
> 个人单机版 Web 热量管理工具，核心目标是帮用户**制造热量缺口**——基于用户的基础代谢和目标缺口，动态计算每日饮食预算，通过记录饮食摄入来监控是否处于缺口状态。

### 1.2 核心概念
```
热量缺口 = 总消耗（基础代谢 + 活动消耗）- 饮食摄入

  ↓                     ↓                      ↓
自动计算               用户设置               用户记录
（BMR 公式）          （活动系数）            （食物+克数）
```

### 1.3 用户画像
| 角色    | 描述                       |
| ----- | ------------------------ |
| 减脂用户  | 有明确的减重目标，需要每日保持热量缺口来逐步减脂 |
| 体重管理者 | 希望维持当前体重，需要热量摄入 ≈ 总消耗    |
| 增肌用户  | 需要热量盈余（反向缺口），增重的同时控制质量   |

### 1.4 核心目标
1. **自动算预算**：基于个人信息自动计算每日饮食预算，而非手动硬填一个数字
2. **缺口可视化**：每餐记录的摄入 vs 预算 = 当前缺口多少，一目了然
3. **趋势可追踪**：日/周/月维度的缺口达成情况，知道自己是否走在正确轨道上

| 模式 | 原理 | 优劣 |

## 3. 业务规则

### 3.1 热量缺口核心模型

```
BMR（基础代谢）
   ↓ × 活动系数
TDEE（每日总消耗）
   ↓ − 目标缺口
每日饮食预算（可摄入热量）

缺口状态：
  已摄入 < 预算     → ✅ 在缺口内（减脂中）
  已摄入 = 预算     → ⚖️ 刚好持平（维持体重）
  已摄入 > 预算     → ⚠️ 超出预算（缺口关闭）
```

> **用户只设置"目标缺口"和"个人信息"，系统自动算预算。** 用户不用猜自己每天该吃多少。

### 3.2 基础代谢 BMR 计算（Mifflin-St Jeor 公式）

```
男性：BMR = 10 × 体重(kg) + 6.25 × 身高(cm) − 5 × 年龄 + 5
女性：BMR = 10 × 体重(kg) + 6.25 × 身高(cm) − 5 × 年龄 − 161

示例（男性，75kg, 175cm, 30岁）：
  BMR = 10×75 + 6.25×175 − 5×30 + 5
      = 750 + 1093.75 − 150 + 5
      = 1698.75 千卡/天
```

### 3.3 活动系数

|   档位    | 描述          |  系数   | 适用场景      |
| :-----: | ----------- | :---: | --------- |
| 1 - 久坐  | 几乎不运动       |  1.2  | 办公室工作、无运动 |
| 2 - 轻度  | 每周运动 1-3 天  | 1.375 | 偶尔散步/轻度运动 |
| 3 - 中度  | 每周运动 3-5 天  | 1.55  | 规律运动      |
| 4 - 高度  | 每周运动 6-7 天  | 1.725 | 每天高强度训练   |
| 5 - 极高度 | 体力劳动 + 每天训练 |  1.9  | 运动员/体力工作者 |

```
TDEE（每日总消耗）= BMR × 活动系数
```

### 3.4 目标缺口与预算计算

| 规则       | 说明                                          |
| -------- | ------------------------------------------- |
| 预算公式     | `每日预算 = TDEE − 目标缺口`                        |
| 目标缺口     | 用户设置（默认 −300 千卡/天），可随时修改                    |
| 减脂建议     | −300 千卡/天 ≈ 每周减 0.3kg；−500 千卡/天 ≈ 每周减 0.5kg |
| 安全下限     | 预算 ≥ BMR × 0.8（过低则警告）                       |
| 缺口为正（增肌） | 用户也可设置正缺口（盈余），如 +200 千卡/天                   |
| 预算更新时机   | 修改个人信息/活动系数/目标缺口后 → 实时重算                    |

#### 完整计算示例

```
用户：男性，当前 80kg，目标 75kg，计划 3 个月（90 天）
      175cm，30岁，中度运动

Step 1 - 每日缺口：
  总减重 = 80 − 75 = 5 kg
  总缺口 = 5 × 7700 = 38500 千卡
  每日缺口 = 38500 ÷ 90 ≈ 428 千卡/天

Step 2 - BMR：
  BMR = 10×80 + 6.25×175 − 5×30 + 5 = 1748.75 千卡

Step 3 - TDEE：
  TDEE = 1748.75 × 1.55 = 2710 千卡

Step 4 - 每日预算：
  预算 = 2710 − 428 = 2282 千卡

→ 每日吃 ≤ 2282 千卡，持续 90 天，预计减重 5 kg ✅
```

### 3.5 直接填写缺口（高级模式）

部分用户可能想跳过目标计算，直接控制每日缺口：

| 模式 | 操作 | 适用人群 |
|------|------|---------|
| **目标模式**（推荐） | 填目标体重 + 目标周期，系统自动算 | 大多数用户 |
| **高级模式** | 直接填写每日缺口数值（如 −300） | 有经验的健身者 |
| **维持模式** | 目标体重 = 当前体重 → 缺口 = 0 | 体重维持者 |
| **增肌模式** | 填写增重目标 → 系统自动算盈余 | 增肌用户 |

> 设置页提供"切换至高级模式"开关，默认目标模式。

### 3.6 今日缺口状态判定

| 条件 | 状态 | 颜色 | 含义 |
|------|------|:----:|------|
| 已摄入 ≤ 预算 | ✅ 在缺口内 | 🟢 绿色 | 继续保持 |
| 预算 < 已摄入 ≤ TDEE | ⚠️ 接近上限 | 🟡 黄色 | 缺口在缩小，注意晚餐 |
| 已摄入 > TDEE | ❌ 超支 | 🔴 红色 | 今日无缺口，热量盈余 |

### 3.7 缺口达成率统计

```
单日判定：已摄入 ≤ 预算 → 缺口达标日 ✅
周达成率 = 本周达标天数 / 7 × 100%
月达成率 = 本月达标天数 / 本月已过天数 × 100%

示例：
  本周 7 天中 5 天在缺口内
  → 周达成率 = 5/7 = 71.4%
```

### 3.8 进度预测与追踪

```
初始设定：目标减重 5kg / 90 天 / 每日缺口 428 千卡

理论进度（每 30 天）：
  D30: 累积缺口 428×30 = 12840 千卡 → 预计减重 1.67 kg
  D60: 累积缺口 428×60 = 25680 千卡 → 预计减重 3.34 kg
  D90: 累积缺口 428×90 = 38500 千卡 → 预计减重 5.00 kg

实际追踪：
  每日记录实际体重 → 对比理论曲线 → 提示"偏快/正轨/偏慢"

延迟影响：
  一周内有 2 天超标 → 周达成率 = 5/7 = 71.4%
  实际减重速度 ≈ 理论 × 周达成率
```

### 3.9 体重预测

```
每累积 -7700 千卡缺口 ≈ 减少 1kg 脂肪

按每日缺口 428 千卡，每周累积缺口 = 428 × 7 = 2996 千卡
预计每周减重 ≈ 2996 / 7700 ≈ 0.39 kg
预计每月减重 ≈ 1.67 kg
```

> 注：以上为理论估算，实际减重受水分、肌肉占比等多种因素影响，仅供参考。

|------|------|
| 拆分规则 | 用逗号/顿号/"和"/"还"/"加"等关键词拆分多条食物 |
| 餐别推断 | 根据时间 + 关键词推断：含"早"→早餐，"午/中"→午餐，"晚"→晚餐，其余→加餐 |
| 标准份量 | 基于内置常见食物的标准份量表（一碗面≈400g 等） |
| 食物匹配 | 关键词匹配内置食物库，匹配不到则标记"未知食物"，提示用户手动选择 |
| 置信度标记 | 规则引擎结果统一标记 `confidence: 'medium'` |
| 模糊表述 | "一些""一点""少量"→按标准份量的 50% 估算并标低置信度 |

#### 大模型模式（需配置 API）

| 规则 | 说明 |
|------|------|
| 调用方式 | 将用户描述 + 系统提示词（system prompt）发送至用户配置的大模型 API |
| 输出格式 | LLM 返回 **结构化 JSON**，由前端解析后展示结果卡片 |
| 典型 Prompt | 见下方 Prompt 模板 |
| 模型要求 | 支持 JSON mode 或 function calling 的模型（GPT-4o / DeepSeek / Qwen 等） |
| 超时处理 | API 请求超时（默认 10 秒）→ 降级为规则引擎模式 |
| 失败降级 | API 请求失败/返回格式异常 → 降级为规则引擎模式，提示用户"大模型估算失败，已切换回本地模式" |
| 成本控制 | 每次估算请求限制输出 max_tokens=512，控制 API 调用成本 |

#### API 配置要求

| 配置项 | 类型 | 说明 | 默认值 |
|--------|------|------|--------|
| `apiEndpoint` | string | API 请求地址 | 空（用户填写） |
| `apiKey` | string | API 密钥 | 空（用户填写） |
| `modelName` | string | 模型名称（如 deepseek-chat、gpt-4o-mini） | 空（用户填写） |
| `mode` | `'rule' \| 'llm' \| 'hybrid'` | 估算模式 | `'rule'`（未配置 API 时）/ `'hybrid'`（已配置 API 时） |

> **安全说明**：API Key 仅存储在 LocalStorage，不上传任何第三方服务器，用户自担使用风险。

#### 大模型 Prompt 模板

```json
{
  "system": "你是一个食物热量估算助手。用户会描述一餐吃了什么，你需要：
  1. 将描述拆分为多个独立食物条目
  2. 推断每项食物的克数（基于标准份量常识）
  3. 匹配食物名称（精确匹配，不要编造）
  4. 返回结构化 JSON

  输出格式要求（仅返回 JSON，不要多余文字）：
  {
    \"mealType\": \"早餐|午餐|晚餐|加餐\",
    \"items\": [
      {
        \"foodName\": \"食物名称\",
        \"grams\": 克数,
        \"confidence\": \"high|medium|low\",
        \"note\": \"备注（如有特殊说明）\"
      }
    ]
  }

  注意事项：
  - 克数基于中国常见餐饮标准份量估算
  - 如不确定可标注 confidence: low

  "user": "中午吃了碗兰州牛肉面，加了个卤蛋，还喝了杯可乐"
}
```
